1. 环境变量配置

.env ENV · 当前激活

位于项目根目录，由 backend/app/config.py 的 Pydantic Settings 自动读取，覆盖默认值。

关键配置项

  • SECRET_KEY123456 ⚠️ 弱密钥
  • ENCRYPTION_KEYUlKNz_nNZE… Fernet 密钥
  • MAX_FILE_SIZE_MB50
  • ALLOWED_ORIGINSlocalhost:5173, :3000

对工作逻辑的影响

  • SECRET_KEY：用于 JWT 签名，决定 Token 安全性
  • ENCRYPTION_KEY：用 Fernet 对称加密敏感数据（如 API Key）
  • ALLOWED_ORIGINS：控制 CORS，仅允许列表内源访问 API
  • MAX_FILE_SIZE_MB：限制上传文件大小，防止资源耗尽

对项目结构的影响

  • 通过 pydantic-settings 注入全局 settings 对象
  • 所有模块通过 from app.config import settings 获取配置，统一管理

对部署的影响

  • 此文件被 .gitignore 排除，不会提交到仓库
  • Docker 容器内由 entrypoint.sh 覆盖密钥，此文件仅用于本地开发
⚠️ 安全风险：SECRET_KEY=123456 极度不安全，任何部署到公网的实例都应替换为强随机密钥。生产环境应通过 entrypoint.sh 的 /secrets/app.env 机制自动生成。
.env.example 模板

配置模板，供开发者初始化本地 .env 文件时参考，揭示了所有可注入的配置维度。

完整配置维度（超出 .env 现值）

  • DATABASE_URL默认 sqlite+aiosqlite:///./xiaoman.db，可替换 PostgreSQL
  • MILVUS_HOST / PORT向量数据库连接配置，USE_MILVUS=false 时忽略
  • USE_MILVUSfalse → 使用 ChromaDB；true → 使用 Milvus
  • EMBED_MODELtext2vec-base-chinese，控制嵌入模型选择
  • UPLOAD_DIR文件上传存储路径
  • FEISHU_APP_ID / SECRET / TOKEN飞书机器人集成三元凭证

2. Python 后端配置

backend/requirements.txt Python 依赖

核心框架与服务层

  • fastapi==0.115.0HTTP/WebSocket 框架
  • uvicorn[standard]ASGI 服务器（含 websockets）
  • sqlalchemy[asyncio]异步 ORM，操作 SQLite
  • aiosqliteSQLite 异步驱动
  • pydantic / pydantic-settings数据验证与配置管理

AI / 向量检索层

  • litellm==1.50.0统一调用 100+ LLM 提供商
  • chromadb>=0.4.22本地向量数据库（RAG）
  • langchain / langchain-community文档分割、嵌入流水线
  • mcp>=1.0.0Model Context Protocol 工具调用

文档处理层

  • pypdf2PDF 解析
  • python-docxWord 文档解析
  • python-pptxPPT 解析
  • Pillow图像处理（OCR 前处理）
  • python-multipart文件上传解析

网络与安全层

  • httpx + httpx-sse异步 HTTP + SSE 流式请求
  • duckduckgo-search网络搜索工具
  • python-jose[cryptography]JWT 签发与验证
  • bcrypt密码哈希
  • cryptographyFernet 数据加密
  • lark-oapi飞书 SDK

对部署的影响

  • 依赖均已精确锁版本（或设置下限），有利于可重现构建，但 langchain>=0.1.0 等宽松约束在未来可能引入破坏性变更
  • chromadb 会在首次运行时下载模型文件，镜像体积较大（可能超过 2GB），需考虑层缓存
  • 构建时使用阿里云 pip 镜像（见 Dockerfile），可规避国内网络问题
  • 无 torch 直接依赖，通过 langchain-community 间接拉取，CI 镜像构建时间较长
backend/app/config.py Pydantic Settings

项目的配置核心。使用 Pydantic BaseSettings 将环境变量自动映射为类型安全的 Python 属性，所有模块从此处导入 settings 单例。

关键默认值

  • database_url绝对路径：backend/xiaoman.db
  • chroma_persist_dirbackend/chroma_data/
  • upload_dirbackend/uploads/
  • use_milvusFalse → 默认 ChromaDB
  • embed_modeltext2vec-base-chinese
  • env_file指向项目根 .env

路径解析机制

  • 使用 __file__ 的 绝对路径向上回溯 找到项目根目录，与 CWD 无关
  • 保证在 Docker 容器（WORKDIR /app）或本地任意目录运行时路径一致
  • extra="ignore"：容忍 .env 中未声明的变量，不抛错

对工作逻辑与结构的影响

  • 单例模式：模块顶层 settings = Settings() 在进程启动时执行一次，后续 import 复用同一对象，避免重复读取 .env
  • 向量数据库切换：use_milvus 控制运行时选择 ChromaDB（轻量本地）或 Milvus（分布式），无需改代码
  • CORS 列表：allowed_origins 直接传入 FastAPI CORSMiddleware，控制哪些前端源可调用 API

3. React 前端配置

frontend/package.json Node 依赖

核心运行时依赖

  • react + react-dom ^18.3UI 框架
  • antd ^5.21Ant Design 组件库
  • axios ^1.7HTTP 客户端，调用后端 API
  • react-router-dom ^6.27SPA 路由
  • zustand ^5.0全局状态管理
  • react-markdown + remark-gfm渲染 AI 回复的 Markdown
  • dayjs ^1.11日期格式化

构建工具链（devDependencies）

  • vite ^5.4构建工具与开发服务器
  • @vitejs/plugin-reactBabel 快速刷新
  • typescript ^5.6类型检查
  • tailwindcss ^3.4原子化 CSS
  • postcss + autoprefixerCSS 后处理

构建脚本说明

  • dev启动 Vite 开发服务器（:5173）
  • buildtsc && vite build → 类型检查后构建
  • preview预览生产构建产物
  • lintESLint 检查（目前 0 警告容忍）

对部署的影响

  • "type": "module"：整个项目使用 ESM，配置文件（vite/tailwind）需 export default
  • 无锁文件提交：docker-compose 中每次 npm install 会联网拉取，建议改为多阶段构建
  • antd 5.x 体积较大，需依赖 Vite 的 tree-shaking 控制产物大小
frontend/vite.config.ts Vite 配置

前后端通信的关键桥梁。开发模式下通过 反向代理 将 /api 请求转发到后端，彻底规避 CORS 问题。

关键配置项

  • server.port5173
  • server.host0.0.0.0（局域网可访问）
  • proxy /api → targethttp://localhost:8000
  • changeOrigintrue（修改请求 Host 头）
  • resolve.alias @→ ./src（路径别名）

对工作逻辑的影响

  • 前端代码中 axios.get('/api/...') 不需要写绝对地址，开发/生产保持一致
  • @ 别名让 import 路径干净，无需 ../../../ 层层回溯
  • host: '0.0.0.0' 使容器内的 Vite 可被宿主机 / 其他容器访问

注意：生产环境代理失效

  • Vite proxy 仅在 npm run dev 时有效。生产构建后，/api 请求会直接发往浏览器所在域，需要 Nginx 或其他反向代理承担此职责
  • 当前 docker-compose 使用的是开发模式（npm run dev），尚无 Nginx 层，生产部署时需补充
frontend/tsconfig.json & tsconfig.node.json TypeScript

主要编译选项

  • targetES2020（现代浏览器）
  • stricttrue（严格类型检查）
  • noEmittrue（TS 不生成 JS，由 Vite 处理）
  • jsxreact-jsx（无需手动 import React）
  • moduleResolutionbundler（配合 Vite）
  • paths @/*src/*（与 Vite alias 对齐）

对工作逻辑的影响

  • strict: true 要求显式类型，减少运行时错误但增加开发摩擦
  • noUnusedLocals: false 允许未使用变量，开发体验友好
  • tsconfig.node.json 单独为 Vite 配置文件（Node 环境）提供编译上下文，避免 DOM 类型污染
frontend/tailwind.config.ts & postcss.config.js CSS 工具链

关键配置

  • content扫描 src/**/*.{ts,tsx}，tree-shake 未使用的类
  • preflight: false禁用全局 CSS Reset，与 Ant Design 共存
  • colors.primary自定义蓝色主题色梯度
  • autoprefixer自动添加浏览器前缀

对工作逻辑的影响

  • preflight: false 是与 Ant Design 共存的关键：若启用 preflight，antd 组件样式会被 Tailwind 的 CSS Reset 重置破坏
  • 自定义 primary 色系需要与 antd 的 theme.token 中的 colorPrimary 手动对齐，否则会有色彩不一致

4. 容器化配置

backend/Dockerfile Docker 镜像

构建步骤分析

  • 基础镜像python:3.11-slim（精简版，减少体积）
  • WORKDIR/app
  • APT 源替换为阿里云镜像（国内加速）
  • pip 源阿里云 PyPI 镜像
  • EXPOSE8000
  • ENTRYPOINT/app/entrypoint.sh

对部署的影响

  • 层缓存优化：先 COPY requirements.txt 再 pip install，代码变更不触发重新安装依赖
  • 无多阶段构建：build-essential 等编译工具留在最终镜像，增大约 200MB；可用多阶段构建优化
  • 无健康检查：建议添加 HEALTHCHECK 指令
  • 挂载了整个 ./backend 目录（见 compose），开发时代码变更实时同步但生产需去掉挂载
backend/entrypoint.sh Shell 脚本

容器启动入口，实现了 "首次运行自动生成密钥" 的 Secrets 引导机制。

执行逻辑

  • 检查 /secrets/app.env 是否存在
  • 若不存在 → 用 Python 生成 SECRET_KEY（urlsafe token）和 ENCRYPTION_KEY（Fernet）写入文件
  • set -a; source /secrets/app.env; set +a → 将密钥注入当前进程环境
  • exec uvicorn app.main:app --host 0.0.0.0 --port 8000 → 替换 shell 进程（无守护进程）

对部署的影响

  • /secrets 目录映射到 Docker 卷 xiaoman_secrets，密钥跨容器重启持久化
  • 密钥只生成一次，容器替换不丢失密钥，用户 JWT 不失效
  • 本地 .env 的密钥会被 /secrets/app.env 的值覆盖（source 后赋值）
  • 若 xiaoman_secrets 卷被删除，密钥重新生成，所有 Token 失效
docker-compose.yml 容器编排

服务拓扑

  • backendbuild ./backend，端口 8000:8000
  • frontendnode:20-alpine，端口 5173:5173
  • 网络xiaoman-net（bridge），服务间用容器名通信
  • 卷 xiaoman_uploads持久化用户上传文件
  • 卷 xiaoman_secrets持久化自动生成的密钥

关键配置项

  • ALLOWED_ORIGINS* ⚠️ 容器中放开所有来源
  • DATABASE_URLsqlite:///./xiaoman.db
  • frontend commandnpm install && npm run dev --host
  • depends_onfrontend 等待 backend 启动
  • restartunless-stopped（backend）

问题与改进点

  • ALLOWED_ORIGINS=*：在容器环境中完全放开 CORS，与 config.py 的限制列表矛盾，生产环境应指定具体域名
  • 无 .env 文件注入：compose 未声明 env_file，飞书等敏感配置未传入容器，需手动补充
  • 前端每次启动都 npm install：生产应改为多阶段构建静态文件 + Nginx 服务
  • 无 healthcheck / depends_on condition：frontend 可能在 backend 就绪前就发出请求
  • 无 Milvus 服务定义：若 USE_MILVUS=true，需手动另外启动 Milvus，compose 文件缺少该服务

关键排除规则

  • .env防止密钥泄漏到仓库
  • *.db / *.sqlite3数据库文件不入库
  • uploads/用户上传文件不入库
  • backend/chroma_data/向量数据库持久化目录
  • node_modules/前端依赖不入库
  • dist/构建产物不入库
  • .claude/Claude Code 本地会话记录

对项目结构的影响

  • .env.example 被提交、.env 被排除 → 正确的机密管理模式
  • chroma_data/ 排除意味着向量数据库数据需要在部署后重新构建索引
  • 缺少 .dockerignore：Docker build context 会包含 node_modules、chroma_data 等大型目录，拖慢构建速度

优先改进建议

安全类（高优先）

  • 将 .env 中的 SECRET_KEY=123456 替换为强随机密钥
  • compose 中 ALLOWED_ORIGINS=* 改为具体域名
  • compose 中添加 env_file: .env 以传入飞书等凭证

部署类（中优先）

  • 添加 .dockerignore 排除 node_modules / chroma_data
  • 添加 Nginx 服务处理生产环境的 /api 反向代理
  • 前端改为多阶段构建 + Nginx 服务静态文件
  • compose 添加 healthcheck 和 depends_on condition: healthy

工程类（低优先）

  • 添加 .eslintrc 明确前端 lint 规则
  • 添加 Makefile 统一 dev/build/deploy 命令
  • 锁定 langchain 等宽松版本依赖，添加 pip freeze 锁文件
