## 阶段一：知识库 - 文件上传解耦

**目标**：修复上传状态误报问题，将文件上传与 chunk 分块解耦为两个独立的异步流程。

### 任务

1. **梳理现有上传流程**：找到文件上传接口和 chunk 分块任务的代码，理解当前耦合点。
2. **解耦上传与分块**：
   - 文件上传接口只负责接收文件、存储到本地文件系统，成功后立即返回 `200`，同时将分块任务投入后台队列（如 Celery / BackgroundTasks / asyncio task）。
   - 前端上传状态只跟踪"文件是否成功写入磁盘"，不再等待分块完成。
3. **分块任务状态追踪**：为每个文件增加分块状态字段（`pending` / `processing` / `done` / `failed`），前端可通过轮询或 WebSocket 展示实时进度。
4. **分块失败错误输出**：分块任务捕获所有异常，将完整错误堆栈写入日志，并将状态更新为 `failed`，附带可读的错误描述字段，前端展示该描述。

## 阶段二：知识库 - 文档图片与表格支持

**目标**：让知识库在 chunk 阶段能提取并索引文档中的图片和表格，LLM 回答时由 LLM 自己决定引用哪些图片。

### 任务

#### 2.1 文档解析增强

1. 解析文档时（PDF / Word 等），识别并提取所有图片和表格。
2. 每张图片/表格分配唯一语义 ID，格式为 `[IMG_{文件名缩写}_{序号}_{简短描述}]`，例如 `[IMG_report_001_季度销售趋势图]`。
3. 调用视觉模型（vision）对每张图片生成自然语言描述，描述存入 metadata，图片文件存到本地文件系统，记录存储路径。
4. 表格转为 Markdown 格式后作为独立 chunk 存储。

#### 2.2 Chunk 关联图片

1. 文字 chunk 存储时，记录该段落所关联的图片 ID 列表（解析阶段根据图片在文档中的位置与段落做关联）。
2. 图片 metadata 单独存储，包含字段：`id`、`description`、`local_path`、`source_doc`、`page_num`。

#### 2.3 LLM 回答时图片引用（LLM 驱动）

1. 检索召回相关 chunk 后，提取这些 chunk 关联的所有图片 ID 和描述，拼入给 LLM 的 prompt，格式如下：
   ```
   文档内容：
   {召回的chunk文本，其中含 [IMG_xxx] 占位}

   可用图片（仅引用与回答相关的）：
   [IMG_xxx] - {图片描述}
   [IMG_yyy] - {图片描述}
   ```
2. System prompt 中明确指示：如回答涉及某图片内容，在对应位置插入该图片 ID 标记；不相关的图片不引用。
3. LLM 返回后，用正则提取所有 `[IMG_xxx]` 标记，校验 ID 是否存在于 metadata 中（过滤幻觉），读取本地图片文件，组装返回结果：
   ```json
   {
     "answer": "回答文本，含 [IMG_xxx] 标记",
     "images": [
       { "id": "IMG_xxx", "description": "...", "base64": "..." }
     ]
   }
   ```

#### 2.4 前端渲染

1. 对话界面解析返回的 `images` 列表，将答案文本中的 `[IMG_xxx]` 替换为对应图片的内联展示（`<img>` 或图片卡片组件）。

## 阶段三：知识库 - Excel 格式支持

**目标**：上传 `.xlsx` / `.xls` 文件时能正常解析并入库。

### 任务

1. 在文档解析模块中增加 Excel 解析器（推荐 `openpyxl` 或 `pandas`）。
2. 解析策略：
   - 每个 Sheet 作为独立文档单元。
   - 每张 Sheet 按行/块切分为 chunk，保留表头信息，每个 chunk 转为 Markdown 表格格式。
   - 如果 Sheet 包含图表，用 vision 模型提取图表描述，同阶段二处理。
3. 在文件类型判断处注册 `.xlsx` / `.xls` 扩展名，路由到新解析器。
4. 测试：上传一个包含多 Sheet 和图表的 Excel 文件，验证 chunk 内容和检索效果。

## 阶段四：飞书 - 响应速度与稳定性修复

**目标**：解决飞书机器人响应慢、群聊 @ 不回复两个核心问题。

### 任务

#### 4.1 响应速度优化

1. 找到长连接消息接收和回复的代码，测量从接收消息到调用 LLM 再到发送回复的各环节耗时，在日志中记录每个关键节点的时间戳。
2. 针对瓶颈点优化：
   - 如果是 LLM 调用慢：开启流式输出（streaming），先发送一条"正在思考…"的占位消息，LLM 回答完成后更新该消息。
   - 如果是知识库检索慢：检查向量检索是否有索引，是否存在 N+1 查询。
   - 如果是消息处理串行阻塞：改为异步并发处理。

#### 4.2 群聊 @ 不回复修复

1. 在飞书自建应用后台确认已开启"群聊消息"事件订阅权限（代码层面打印接收到的原始事件，检查群聊 @ 消息的 `event_type` 和 `mention` 字段是否被正确识别）。
2. 找到消息路由逻辑，检查群聊场景下 @ 机器人的判断条件（通常需要检查 `mentions` 列表中是否包含机器人自身的 `open_id`）。
3. 修复判断逻辑，确保群聊 @ 时触发与私聊相同的回复流程。
4. 测试：在群聊中 @ 机器人发送消息，验证能正常回复。

#### 4.3 飞书聊天默认使用所有知识库

1. 找到飞书对话调用 LLM/检索的入口，定位知识库选择逻辑。
2. 将飞书渠道的默认知识库范围设置为"全部"（而不是某个特定知识库或空）。
3. 如果产品上需要可配置，增加飞书渠道的知识库范围配置项。

## 阶段五：对话 - 多模态输入支持

**目标**：对话界面支持上传图片和语音输入，结果支持图片渲染。

### 任务

#### 5.1 图片上传输入

1. 前端对话输入框增加图片上传按钮，支持选择或拖拽图片（`jpg` / `png` / `webp`），预览后随消息一起发送。
2. 后端接口接收图片，转为 base64 后拼入发送给 LLM 的消息（`image` 类型 content block），LLM 使用 vision 能力理解图片内容后回答。
3. 对话历史中保存图片消息的存储路径，展示时加载本地图片。

#### 5.2 语音输入

1. 前端增加录音按钮，录制完成后将音频文件上传到后端。
2. 后端调用语音转文字服务（接入 Whisper API ），将转录文本作为用户消息发送给 LLM。
3. 前端在发送前展示转录文本，允许用户编辑后再发送。

#### 5.3 对话结果图片渲染

1. 统一对话消息的返回格式，增加 `images` 字段（与阶段二保持一致）。
2. 前端消息渲染组件：检测消息中的 `[IMG_xxx]` 标记，替换为图片组件（支持点击放大）；检测 `images` 列表，在消息底部展示关联图片卡片。
3. 确保图片加载失败时有降级展示（显示图片描述文本）。

## 阶段六：通用 - 前后端日志系统

**目标**：前后端统一建立日志功能，自动管理日志文件大小和生命周期。

### 任务

#### 6.1 后端日志

1. 配置日志系统（使用标准库 `logging` + `RotatingFileHandler` 或 `TimedRotatingFileHandler`）：
   - 日志目录：`logs/`（项目根目录下）。
   - 按日期滚动：每天生成一个日志文件，保留最近 7 天。
   - 单文件大小上限：5MB，超出后自动创建新文件（序号递增）。
   - 日志目录总大小上限：1GB，超出后从最旧的文件开始删除，直到低于上限。
2. 日志级别：`INFO` 记录正常流程关键节点，`ERROR` 记录异常和堆栈，`WARNING` 记录可恢复的异常情况。
3. 统一日志格式：`{timestamp} [{level}] [{module}] {message}`。
4. 将阶段一中分块失败的错误、阶段四中飞书消息处理耗时等都接入此日志系统。

#### 6.2 前端日志

1. 前端增加日志收集模块，捕获：
   - 用户操作关键事件（上传文件、发送消息、切换知识库等）。
   - 接口请求的请求/响应状态和耗时。
   - JS 运行时错误（`window.onerror` / `unhandledrejection`）。
2. 日志批量上报到后端日志接口（`POST /api/logs`），后端将前端日志写入独立的 `logs/frontend/` 目录，遵循相同的滚动规则。
3. 前端本地也用 `localStorage` 暂存最近 100 条日志，网络失败时下次重试上报。
